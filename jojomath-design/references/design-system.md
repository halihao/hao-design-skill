# JoJoMath 设计系统

这份文档定义可复用的 UI 语言。当前先用设计变量名称和规则占位，具体数值后续需要从官方 Figma 样式或已上线产品样式代码中提取。

## 颜色变量

| 变量 | 用途 | 取值 | 说明 |
| --- | --- | --- | --- |
| `brand.primary` | 主操作、核心品牌时刻 | 待补充 | 从 Figma 或品牌资产确认。 |
| `brand.secondary` | 次级强调色或辅助区域 | 待补充 | 不要抢主操作的层级。 |
| `semantic.success` | 答对、完成、进步 | 待补充 | 在浅色背景上必须可读。 |
| `semantic.warning` | 提示、注意、部分风险 | 待补充 | 学生端使用时不要产生惩罚感。 |
| `semantic.error` | 答错、阻断错误 | 待补充 | 需要配合解释性文案。 |
| `surface.base` | 页面主背景 | 待补充 | 必须保证数学内容可读。 |
| `text.primary` | 主文字和数学内容 | 待补充 | 最高对比度文字。 |
| `text.secondary` | 辅助标签和说明文案 | 待补充 | 不要用于核心题干。 |

## 字体与排版

- **数学内容**：待补充：字体、字号范围、行高和公式渲染规则。
- **学生端标题**：待补充：亲和但清楚；紧凑学习工具里不要使用过大的标题。
- **家长端/报告标题**：待补充：更克制、数据优先、便于扫读。
- **营销标题**：待补充：直接表达收益，足够短，适配移动端和海报。
- **数字和进度**：待补充：分数、连续学习、正确率、用时、排名等数字的展示规则。

## 布局与间距

- 练习卡片、答题区域、控件、进度条和报告模块要有稳定尺寸。
- 主操作要放在用户自然扫视路径中。
- 不要做卡片套卡片，除非内部卡片是重复项目或真正的子模块。
- 待补充：补充间距体系、页面最大宽度、移动端断点和网格规则。

## 组件

后续加入 Figma 或产品案例后，记录每个组件的标准变体。

- **主按钮**：待补充：颜色、圆角、尺寸、图标规则、禁用和加载状态。
- **次按钮**：待补充：使用场景和视觉层级。
- **题目容器**：待补充：题干布局、图片/音频支持、长内容处理。
- **答题输入**：待补充：选择题、文本/数字输入、键盘、手写或公式输入。
- **反馈模块**：待补充：答对、答错、提示、解析、重试和完成状态。
- **进度指示器**：待补充：课程进度、掌握度、连续学习和报告进度。
- **奖励或庆祝**：待补充：使用时机、强度、动画/图片风格。
- **报告卡片**：待补充：家长端指标、洞察和建议的展示模式。

## 图标与插画

- 图标用于澄清动作、状态和工具，不要只作为无意义装饰。
- 学生端插画可以更温暖、更有趣，但要服务学习动机。
- 家长端视觉要更克制，更强调证据和可信度。
- 待补充：补充官方图标库、插画风格、角色/吉祥物使用规则和禁用风格。

## 可访问性与质量

- 文字、数学内容和控件必须保持足够对比度。
- 不要只依靠颜色表达对错或错误状态。
- 长公式、翻译文案和移动端布局不能溢出。
- 触控设备上的点击区域要足够舒适。
- 所有例外都要说明产品原因，而不是单纯的视觉偏好。
